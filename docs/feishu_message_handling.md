# 飞书消息处理说明

本文档说明如何处理飞书平台的各种消息类型。

## 事件订阅

本项目使用 **`im.message.receive_v1`** 事件接收所有消息类型。

### 事件说明

**`im.message.receive_v1`** - 接收消息事件
- 包含所有类型的消息
- 通过 `chat_type` 字段区分：
  - `p2p` - 私聊消息
  - `group` - 群聊消息
  - `public` - 公群消息

## 消息处理逻辑

### 1. 私聊消息（p2p）

**处理方式**: 总是响应

- 用户发送给机器人的私聊消息会直接响应
- 不需要判断 @机器人
- 适合一对一的对话场景

### 2. 群聊消息（group）

**处理方式**: 只有 @机器人 时才响应

- 需要判断消息的 `mentions` 列表
- 精确匹配机器人的 `user_id`
- 适合群聊中与机器人交互的场景

## bot_user_id 的作用

### 什么是 bot_user_id？

`bot_user_id` 是机器人在飞书平台上的唯一标识符，格式类似 `cli_xxxxxxxxxxxxx`。

### 为什么需要 bot_user_id？

在群聊场景中，需要判断用户是否 @了机器人：

```json
{
  "mentions": [
    {
      "id": {
        "user_id": "cli_xxxxxxxxxxxxx",
        "open_id": "ou_xxxxxxxxxxxxx"
      },
      "name": "Claude 机器人",
      "tenant_key": "xxxxx"
    }
  ]
}
```

只有当 `mentions` 列表中包含机器人的 `user_id` 时，才认为机器人被 @了。

### 如何获取 bot_user_id？

#### 自动提取并保存（推荐）

首次使用时，在群聊中 @机器人，系统会自动：

1. **提取** bot_user_id
2. **保存** 到 `${DEFAULT_SESSION_ROOT}/robot_id.txt` 文件
3. **加载** 下次启动时自动从文件加载

```
✅ 从消息 mentions 中提取到 bot_user_id: cli_xxxxxxxxxxxxx
✅ bot_user_id 已保存到文件: sessions/robot_id.txt
💡 下次启动将自动加载此 ID
```

**无需手动配置！**

#### 手动配置（可选）

如果需要手动配置，可以在 `.env` 文件中添加：

```bash
FEISHU_BOT_USER_ID=cli_xxxxxxxxxxxxx
```

**优先级**：配置文件 > robot_id.txt 文件 > 未设置

### 宽松模式 vs 精确模式

#### 宽松模式（无 bot_user_id）

- **私聊**: 正常响应
- **群聊**: 只要消息有 `mentions` 列表就响应
- **用途**: 首次使用时自动提取 bot_user_id
- **缺点**: 可能在群聊中误响应（@其他人时也会响应）

#### 精确模式（有 bot_user_id）

- **私聊**: 正常响应
- **群聊**: 只有精确匹配 `bot_user_id` 时才响应
- **用途**: 正常使用场景
- **优点**: 精确判断，避免误响应

## 配置建议

### 开发环境

```bash
# .env
FEISHU_APP_ID=your_app_id
FEISHU_APP_SECRET=your_app_secret
# FEISHU_BOT_USER_ID  # 可选，首次使用后填入
```

首次使用时，先在群聊中 @机器人一次，自动提取 `bot_user_id`，然后填入配置文件。

### 生产环境

```bash
# .env
FEISHU_APP_ID=your_app_id
FEISHU_APP_SECRET=your_app_secret
FEISHU_BOT_USER_ID=cli_xxxxxxxxxxxxx  # 必填，避免误响应
```

生产环境建议直接配置 `bot_user_id`，确保精确判断。

## 其他飞书事件类型

飞书还提供了其他消息相关的事件：

### `im:message.group_at_msg:readonly`

- **说明**: 只接收群聊中@机器人的消息
- **权限**: 免审权限
- **优点**: 不需要判断 `mentions`，接收到的消息一定@了机器人
- **缺点**: 需要同时订阅两个事件（私聊用 `receive_v1`，群聊用 `group_at_msg`）

### 本项目选择

本项目选择只订阅 `im.message.receive_v1` 事件，因为：

1. **统一处理**: 一个事件处理所有消息类型
2. **灵活判断**: 可以根据需要调整响应逻辑
3. **简化配置**: 不需要订阅多个事件

如果后续需要优化，可以考虑同时订阅 `im:message.group_at_msg:readonly` 事件，这样可以：
- 减少不必要的消息处理
- 提高性能
- 降低误响应风险

## 表情反应

当AI处理用户消息时，系统会自动添加表情反应以提供状态反馈：

1. **敲键盘表情** (Typing): AI开始处理消息时自动添加
2. **完成表情** (DONE): AI完成所有响应后替换

### 消息引用

AI回复的"思考中..."卡片会引用用户消息（通过`parent_id`），便于在飞书中追踪完整的消息链。

### 容错处理

表情操作失败不会影响AI的主流程。如果无法添加表情，系统会记录日志并继续处理消息。

### 技术实现

表情功能由`FeishuReactionManager`类实现，位于`src/bridges/feishu/reaction_manager.py`。

- 支持添加、删除、替换表情反应
- 状态管理：`_pending_reactions`字典跟踪每个会话的表情信息
- 异常安全：使用try-finally确保状态总是被清理
